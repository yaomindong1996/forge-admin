# 数据集权限控制使用说明

## 1. 完成状态

数据集权限控制已按双层模型实现完成：

- 数据集访问权限：控制哪些用户、角色、组织能看到和使用数据集。
- 数据集行权限：控制同一个数据集在运行时能查询到哪些数据行。

本功能没有改动系统角色授权、菜单权限、`sys_role.data_scope` 的既有语义，也没有改动 `DataScopeInterceptor`。默认兼容历史数据：数据集默认公开，行权限默认关闭。

## 2. 上线前数据库脚本

新环境可直接使用 `src/main/resources/sql/data_tables.sql`。

已有环境按顺序执行：

```sql
-- 1. 数据集访问权限
source src/main/resources/sql/data_dataset_permission_upgrade_20260515.sql;

-- 2. 数据集行权限
source src/main/resources/sql/data_dataset_row_scope_upgrade_20260515.sql;
```

执行后会新增：

- `ai_report_data_dataset.access_mode`
- `ai_report_data_dataset_acl`
- `ai_report_data_dataset_row_scope`

## 3. 权限模型

### 3.1 第一层：访问权限

访问模式：

- `PUBLIC`：公开数据集，保持原有行为，授权列表不生效。
- `PRIVATE`：私有数据集，只允许管理员、创建人、授权主体访问。

授权主体：

- `ROLE`：角色
- `USER`：用户
- `ORG`：组织

权限级别：

- `VIEW`：可查看数据集和元数据。
- `QUERY`：可查询数据集运行时数据，包含 `VIEW` 能力。
- `MANAGE`：可管理私有数据集，包含 `QUERY` 能力。

运行时接口会再次校验权限，不能只依赖列表是否可见。

### 3.2 第二层：行权限

行权限默认关闭。开启后，运行时查询会读取当前登录人的系统数据范围，并拼接参数化 SQL 条件。

当前支持的系统数据范围：

| 系统数据范围 | 运行时行为 |
| --- | --- |
| 全部数据 | 不追加行过滤 |
| 本租户数据 | 配置了租户字段时追加租户过滤 |
| 本组织数据 | 使用组织字段过滤当前用户组织 |
| 本组织及子组织 | 使用组织字段过滤当前用户组织及子组织 |
| 个人数据 | 使用用户字段过滤当前登录用户 |
| 本行政区划数据 | 使用行政区划字段过滤当前用户行政区划范围 |

多角色用户会按系统角色数据范围取最宽权限，保持和现有角色数据权限思路一致。

## 4. 页面配置入口

入口：数据集资产管理台 -> 新增/编辑数据集 -> 第 4 步“权限控制”。

### 4.1 配置访问权限

1. 选择“公开”或“私有”。
2. 如果选择“私有”，点击“添加授权”。
3. 选择主体类型：角色、用户、组织。
4. 选择授权主体。
5. 选择权限级别：查看、查询、管理。
6. 保存数据集。

建议：

- 普通报表消费角色配置 `QUERY`。
- 数据集维护角色配置 `MANAGE`。
- 只需要看元数据的角色配置 `VIEW`。

### 4.2 配置行权限

1. 打开“数据行权限”开关。
2. 根据数据源字段配置字段映射：
   - 租户字段：如 `tenant_id`
   - 组织字段：如 `org_id`、`dept_id`
   - 用户字段：如 `user_id`、`create_by`
   - 行政区划字段：如 `region_code`、`city_code`、`county_code`、`grid_code`
3. 行政区划策略按业务选择：
   - 仅本级
   - 本级及直接下级
   - 本级及所有下级
4. 保存数据集。

字段下拉优先使用数据集已同步字段。单表数据集如果尚未同步字段，可点击“刷新字段”从来源表读取字段作为待选项。

## 5. 典型组织权限配置

假设组织层级为：区公司 -> 盟市 -> 区县 -> 网格。

推荐配置：

| 账号类型 | 系统角色数据范围 | 数据集行权限字段 | 行政区划策略 | 效果 |
| --- | --- | --- | --- | --- |
| 区公司账号 | 全部数据 | 可不配置或配置租户字段 | - | 看全部 |
| 盟市账号 | 本行政区划数据 | `region_code` | 本级及所有下级 | 看本盟市、区县、网格 |
| 区县账号 | 本行政区划数据 | `region_code` | 本级及所有下级或仅本级 | 看所属区县及下级 |
| 网格账号 | 本行政区划数据 | `region_code` | 仅本级 | 看本网格 |

如果业务表拆成 `city_code`、`county_code`、`grid_code` 多个字段，建议优先沉淀一个统一的 `region_code` 字段，行权限会更稳定。

## 6. SQL 数据集写法

SQL 数据集启用行权限时，必须在 SQL 中显式放置：

```sql
/*DATA_SCOPE*/
```

推荐写法：

```sql
SELECT
  id,
  region_code,
  amount,
  create_time
FROM order_summary
WHERE status = 1
  /*DATA_SCOPE*/
```

运行时会把占位符替换成参数化条件，例如：

```sql
AND `region_code` IN (:__scopeRegion0, :__scopeRegion1)
```

不要把占位符放在聚合后的外层 SQL，除非外层仍保留了用于过滤的权限字段。

## 7. 单表数据集行为

单表数据集不需要写占位符。启用行权限后，系统会在自动生成的查询后追加条件：

```sql
SELECT ... FROM table_name WHERE 1 = 1
AND `region_code` IN (:__scopeRegion0, :__scopeRegion1)
```

## 8. 兼容与注意事项

- 历史数据集 `access_mode` 默认是 `PUBLIC`。
- 未启用行权限的数据集不会追加任何行过滤。
- 私有数据集的列表、详情、元数据、运行时查询都会做后端校验。
- 行权限字段必须是运行时 SQL 可识别的字段或表达式。
- 启用行权限后，如果当前用户的数据范围需要某个字段但数据集未配置该字段，运行时会拒绝查询并提示配置缺失。
- SQL 参数统一走命名参数绑定，不要手动拼接用户输入。

## 9. 常见问题

### 9.1 私有数据集授权后仍看不到

检查：

- 数据集是否已保存为 `PRIVATE`。
- 授权主体是否是当前用户、当前用户角色或当前用户组织。
- 授权级别是否满足入口要求：查询运行时数据至少需要 `QUERY`。

### 9.2 SQL 数据集启用行权限后报占位符缺失

在 SQL 的 `WHERE` 条件区域加入：

```sql
/*DATA_SCOPE*/
```

### 9.3 盟市账号只能看到本级，看不到区县和网格

检查“行政区划策略”是否选择了“本级及所有下级”。

### 9.4 行权限开启后查不到数据

检查：

- 登录用户是否有行政区划、组织或用户属性。
- 数据表中的权限字段值是否与系统组织/行政区划编码一致。
- 行权限字段是否配置到了正确字段。
